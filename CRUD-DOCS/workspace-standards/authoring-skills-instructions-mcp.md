# Как правильно в агентной инженерии создавать, хранить и использовать инструкции, навыки и MCP

> **Это база для ИИ и человека.** Авторитетный над-документ о том, как в этом воркспейсе рождается,
> где лежит и как срабатывает любая **способность** агента — **инструкция**, **навык** или **MCP**.
> Цель: чтобы любая способность была **самодостаточна** и **открываема всеми шестью агентами**
> (Hermes + 5 кодеров), а не «лежала, но не работала». Живёт в `CRUD-DOCS/workspace-standards/`,
> едет к клиенту с клоном, виден на вкладке «Документы». Опирается на
> [`ai-draft-settings.md`](./ai-draft-settings.md) (слой черновиков),
> [`skill-evolution-pipeline.md`](./skill-evolution-pipeline.md) (цикл эволюции),
> [`development-methodology.md`](./development-methodology.md) (общая дисциплина).

---

## §0. Три способности и одно железное правило

Агент усиливается тремя видами способностей:

- **Инструкция** — как агент *думает* (корневой файл личности/правил).
- **Навык (skill)** — воспроизводимая процедура «как сделать X» (markdown, который агент читает в момент нужды).
- **MCP** — *инструмент*: мост-сервер, который агент реально *вызывает* (читает/пишет состояние).

> **🔒 Железное правило самодостаточности.** Проект может работать с **одним-единственным** агентом —
> без Hermes, без памяти, любое подмножество из 6. Поэтому **каждая способность ОБЯЗАНА быть
> самодостаточной на уровне отдельного агента и продублирована во ВСЕ агент-сущности.** Ничто не
> зависит от существования Hermes или конкретного агента. Если способность подключена только к Hermes
> (или одному агенту) — проект с единственным Codex теряет её целиком. Это **баг**, не «частично готово».

Шесть агентов: **Hermes** (мозг/оркестратор) · **claude-code** · **codex** · **gemini-cli** ·
**qwen-code** · **kimi-code** (пять кодеров).

## §1. Канон формата — ОДИН для всех шести (важнейший раздел)

Способность бесполезна, если агент её **не открывает**. Разные агенты читают способности из **разных
мест** и в **разной форме** — поэтому канон фиксируется раз и применяется везде.

### Навык — директория `<name>/SKILL.md` + YAML-frontmatter (НИКОГДА не плоский `<name>.md`)

```
<name>/
  SKILL.md        ← обязателен ровно с этим именем файла
```

```markdown
---
name: <name>                    # обязателен; по нему агент находит навык
description: >                  # триггер-ключи: по ним агент решает «это про меня»
  Что навык делает и КОГДА его звать — «rename my app», «add French», …
metadata:
  hermes:
    tags: [ключевые, слова, для, ранжирования]
---

# <name>

<тело навыка: когда применять, шаги, примеры>
```

**Почему именно так (доказано):** Hermes открывает навыки ТОЛЬКО как файл с именем `SKILL.md` внутри
папки навыка (его загрузчик `agent/skill_utils.py` обходит дерево и матчит строго `SKILL.md`; имя берёт
из frontmatter `name:` или имени папки). **Плоский `<name>.md` он не видит вообще** → выбирает чужой
похожий навык. Пять кодеров и без того читают `.agents/skills/<name>/SKILL.md` с frontmatter — **та же
форма**. Значит один канон закрывает всех шестерых.

| Агент | Где лежит открываемая форма навыка |
|---|---|
| claude-code | `.claude/skills/<name>/SKILL.md` (копия канона) |
| gemini-cli | `.gemini/skills/<name>/SKILL.md` (копия) |
| qwen-code | `.qwen/skills/<name>/SKILL.md` (копия) |
| codex, kimi-code | читают канон `.agents/skills/<name>/SKILL.md` напрямую |
| **Hermes** | `services/hermes-skills/<name>/SKILL.md` (субстрат) → bootstrap `cp -r` → `/root/.hermes/skills/<name>/SKILL.md` |

> **Канон-источник для пяти кодеров — `.agents/skills/<name>/SKILL.md`**; копии/симлинки в
> `.claude/.gemini/.qwen/skills` обслуживают тех, кто читает свою папку. На Windows симлинки хрупки →
> **копии надёжнее.** **Hermes держит СВОЮ копию** в субстрате (`services/hermes-skills/`), потому что
> слот `app/` сменный — субстрат не может зависеть от файлов гостя.

### MCP — мост-сервер + регистрация в КАЖДОМ агенте + реестр

Недостаточно поднять мост-сервер. MCP открываем, только если он зарегистрирован в **MCP-клиенте каждого
агента**:

| Агент | Где регистрируется MCP |
|---|---|
| 5 кодеров | `.mcp.json` слота (и per-agent конфиг, если есть) |
| Hermes | `~/.hermes/config.yaml` → `mcp_servers:` (+ `Authorization: Bearer …`) |
| все | строка в авторитетном каталоге `MCP-REGISTRY.md` |

Мост обязан отвечать на JSON-RPC `initialize` + `tools/list` (без этого Hermes регистрирует 0
инструментов — урок отдельного антипаттерна). Имя инструмента — `<tier>_<area>_<action>_<object>`
(см. §3).

### Инструкция — merge/replace в корневой файл личности агента

| Агент | Корневой файл |
|---|---|
| claude-code | `CLAUDE.md` |
| codex | `AGENTS.md` |
| gemini-cli | `GEMINI.md` |
| qwen-code | `QWEN.md` |
| kimi-code | `KIMI.md` |
| Hermes | `SOUL.md` (личность) + `HERMES.md` (правила проекта) |

## §2. Как СОЗДАЁТСЯ способность — конвейер draft → step → materialize

Создание идёт не «правкой живых файлов вслепую», а через аудируемый слой черновиков
(`/ai-draft-settings`, стандарт [`ai-draft-settings.md`](./ai-draft-settings.md)):

1. **Черновик (предложение).** Любой агент или архитектор кладёт пожелание на `/ai-draft-settings` —
   навыком `propose-new-agent-skill-or-mcp` (самодостаточный HTTP-путь) или MCP
   `owner_draft_create_record` (если есть Hermes/мост). Реальный файл НЕ трогается.
2. **Сбор в шаг.** Пожелания собираются в один шаг разработки (кнопка 🚀 / MCP `owner_draft_send_to_steps`).
3. **Материализация (КЛЮЧЕВОЙ момент).** Агент применяет черновик к **реальному** файлу — и ОБЯЗАН
   писать его в **канон §1** для КАЖДОГО агента. Именно здесь раньше ломалось: навык писали плоско → он
   не открывался Hermes. Формат-контракт — `ai-draft-settings.md` → «Materialization format contract».
4. **Дублирование во все агенты.** Та же способность кладётся в открываемой форме во все шесть мест
   (§1). Навык → во все skills-папки + субстрат Hermes; MCP → во все MCP-клиенты; инструкция → во все
   корневые файлы.

> Полный замысел эволюции способности (исполнение → тест → визуализация → переписывание, звенья 3–7) —
> [`skill-evolution-pipeline.md`](./skill-evolution-pipeline.md). Этот документ — про **форму и хранение**,
> тот — про **цикл доведения до качества**.

## §3. Именование (одно правило для навыков и MCP)

Имя — **4–6 слов, ближе к 6**, человек понимает с ходу (имена читают и люди):

- **Навык** — kebab-case, без тир-префикса: `propose-new-agent-skill-or-mcp`,
  `scaffold-declared-route-into-component-skeleton`.
- **MCP-инструмент** — snake_case, `<tier>_<area>_<action>_<object>`; тир (`public`/`user`/`owner`) —
  первое слово: `owner_app_settings_set_text_value`.

Два кратких слова, которые человек не расшифрует, — плохое имя. Потолок 6 слов не превышать.

## §4. Как способность ИСПОЛЬЗУЕТСЯ

- **Навык** агент **читает** в момент нужды: по `description`/тегам решает «это про меня» → берёт
  процедуру. Поэтому `description` обязан нести **триггер-ключи** запроса («rename», «add language»…).
- **MCP** агент **вызывает** как инструмент. Перед любым **мутирующим** вызовом — обязательный
  переспрос (навык `confirm-before-mutation`, MCP-REGISTRY §8.2): «Правильно ли я вас понимаю…» +
  перечислить действия + ждать подтверждения. Read-only — без переспроса.
- **Инструкция** действует постоянно (личность/правила), её не «зовут».
- **«Словами».** Конечная проверка: владелец формулирует задачу обычным языком → агент сам подбирает
  верный навык, читает состояние нужным MCP, переспрашивает, выполняет, отчитывается. Если навык
  открываем и `description` триггерный — это работает у всех шести, включая Hermes.

### §4.1. Выбор способности на масштабе (грунтовка + раскрытие)

Способностей может быть одна или миллион — выбор НЕЛЬЗЯ настраивать по-навыку (это O(N) правил). Нужен
один принцип, работающий над любым числом (O(1)):

**Грунтовка (машинно).** На любой запрос-действие агент ПЕРВЫМ делом ищет способность в СВОём
инструментарии (у Hermes — `skills_list`/`skill_view` + свои MCP; у кодеров — нативный поиск по навыкам/
инструментам) и действует по найденному. **Не перечислять варианты из общих знаний и НИКОГДА не
предлагать действие, под которое нет инструмента.** Уточнять только если после поиска подходят 2+ ТВОИХ
реальных инструмента — и тогда варианты = эти реальные инструменты. → описание навыка (`description`) =
**индекс поиска** (retrieval), а не дамп всех навыков в промпт; при масштабе — поиск top-k, не
перечисление.

**Раскрытие (человечно).** Когда агент ВЫНОСИТ человеку выбор способности — не оставлять его слепым:
предложить ссылки на изучение всего каталога (человек сразу думает «а что ещё есть? может, лучше?»).
Identity-aware: владельцу/архитектору — живой `/ai-core` (правда этого сервера); публичному пользователю —
публичные витрины проекта (`/{lang}/skills`,`/{lang}/mcp`, сырые доки). Дефолт: ясный запрос → агент
действует сам; раскрытие — лёгкая опция, когда человек выбирает/исследует, не стена на каждом ходу.

**Симметрия:** агент ищет в инструментарии (машина) ↔ даёт человеку тот же каталог ссылкой (человек) =
пара «внутренняя декларация способностей ↔ публичная витрина/дока». Антипаттерн — дженерик-уточнение из
обучающих данных с вариантами, которых агент не умеет (реальный кейс: Hermes на «rename my app» предложил
app-stores/repo вместо App Settings). → шаг 140.

## §5. Чек-лист «способность готова» (критерий успеха)

Способность готова, ТОЛЬКО когда по каждому пункту — да:

- [ ] **Форма по §1** соблюдена (навык = `<name>/SKILL.md` + frontmatter `name:`; MCP = мост + клиенты +
      реестр; инструкция = в корневых файлах).
- [ ] **Продублирована во ВСЕ 6 агентов** в их открываемой форме (правило самодостаточности §0).
- [ ] **Hermes реально открывает:** `find /root/.hermes/skills -maxdepth 2 -name SKILL.md` содержит навык;
      MCP виден в `tools/list`.
- [ ] **Поведенческий тест:** на профильный запрос «словами» агент выбирает ИМЕННО этот навык/MCP (а не
      чужой похожий), переспрашивает на мутацию, выполняет.
- [ ] **Имя по §3**; `description` несёт триггер-ключи.
- [ ] Паритет проверен у **Hermes И** хотя бы одного кодера (одинаковое поведение).

## §6. Антипаттерны (задокументированные тупики)

- **Плоский `<name>.md` навыка** — Hermes не открывает (видит только `<name>/SKILL.md`) → выбирает
  чужой навык. Канон = директория + frontmatter. (`reports/errors/hermes-skills-flat-md-not-discovered.md`.)
- **MCP без `initialize`/`tools/list`** — Hermes регистрирует 0 инструментов.
- **«Зарегистрирован» ≠ «открываем».** Файл лежит, но в неверной форме/месте — это баг, не «почти готово».
- **Централизация на Hermes** — единая точка отказа; проект с одним кодером теряет способность. Запрещено.
- **Правка живых файлов вслепую** вместо конвейера черновиков — теряется аудит эволюции.
