# Мультиязычный контент — живой стандарт стартера

> 📐 **Часть более широкого [стандарта «Контент-движок»](./content-engine.md)** — полной
> co-location-архитектуры (тонкий `page.tsx` + `_components`/`_lib`/`_data`, neutral-каталог
> блоков, авто-обнаружение, удаляемость без сирот и масштабирование на любую новую
> поверхность вроде магазина). Этот файл остаётся узким **рецептом i18n**; как устроена
> вся машина целиком — см. документ «Контент-движок».

> **Это живой стандарт для одной задачи:** как в проекте хранить и переводить контент (новости, блог,
> документацию, любые страницы), чтобы он **масштабировался по построению** — на десятки языков и тысячи
> страниц, без переписывания старого и без костылей в коде. Документ **растёт вместе с разработкой**.
> Написан по-русски (язык авторитетного замысла); английская копия для скачивания на сайте лежит в
> `public/docs/multilingual-content.md`.
>
> **Зачем это в стандартной архитектуре.** Любой реальный проект рано или поздно выходит за один язык.
> Особенно — в Европе, где двуязычность (а часто и три-четыре языка) это не «фича на будущее», а условие
> запуска. Поэтому подход к мультиязычному контенту вынесен в **стандарт стартера** и подкреплён навыком
> агента [`create-multilingual-content-entry`](../../.agents/skills/create-multilingual-content-entry/SKILL.md):
> любой агент проекта умеет завести мультиязычный документ правильно, без изобретения схемы заново.
>
> **Строится на** (папка `CRUD-DOCS/workspace-standards/`):
> [`development-methodology.md`](./development-methodology.md) — мастер-методология ·
> [`shell-component-architecture.md`](./shell-component-architecture.md) — скелет маршрута/страницы ·
> [`site-settings.md`](./site-settings.md) — выбор языков. Оперативный пайплайн агента — `CLAUDE.md §6`.

## §0. Цель и угол зрения

**Цель.** Хранить контент так, чтобы **новый язык = новый файл**, а не переписывание всего; чтобы
**частичный перевод** был штатным (язык можно выкатить с одним переведённым полем); чтобы в коде страниц
**не было выбора по языку** (`если язык == X`). Это снимает три грабли, которые ломаются на росте:
раздувающиеся файлы-простыни, переводы «всё или ничего», и языковые костыли по всему коду.

**Контекст — это часть оптимизации сайта.** Контент-слой — это «процессор обновления сайта»: то, как на
сайте появляются и живут страницы. Удобная навигация и быстрый поиск нужного — половина дела; вторая
половина в том, что **любая оптимизация сегодня идёт рука об руку с поисковой оптимизацией под Google и
под ИИ** (GEO — generative engine optimization). Поэтому схема ниже сразу учитывает: разные языки = разные
URL с `hreflang`, своя SEO-поверхность на язык, и машиночитаемые индексы для ИИ-краулеров.

**Слой.** product / стартер (`CRUD-DOCS/…`) — едет к клиенту с клоном; описывает контент **внутри
развёрнутого рабочего пространства**.

## §1. Документ = co-located папка маршрута (per-document, per-language файлы)

Каждая единица контента — **отдельная co-located папка маршрута**, а не строка в общем растущем массиве:

```
app/[lang]/<tab>/<slug>/        # один документ = одна папка маршрута
  page.tsx                      # тонкий: re-export Entry + generateMetadata из ./_components
  _components/index.tsx         # композиция: createContentPost / createContentPage({ … })
  _data/
    meta.ts        # неделимые на язык поля: slug, дата, теги, автор, изображения
    en.ts          # ПОЛНОЕ тело на языке-базе (обязательно): переводимые поля + блоки + FAQ
    <lang>.ts      # частичный оверрайд: только то, что отличается от базы
    index.ts       # export const data = { meta, en, overrides: { <lang> } }
```

- **`en` — обязательная полная база.** Любой другой язык — частичный оверрайд; чего нет — берётся из базы
  **поштучно** (резолвер `resolveEntry`, общий движок `lib/content`; при версионировании движка путь —
  `lib/content-vN`, side-by-side с пиннингом, см. [`frozen-template-constructor.md`](./frozen-template-constructor.md) §8).
  i18n здесь — это **единообразный аспект** конструктора (Slot B): встраивается одинаково на каждом уровне
  структуры независимо от глубины/источника (там же §5).
- **Новый язык = новый файл** `_data/<lang>.ts` в той же папке. Существующие файлы не трогаются.
- **Новый документ = новая папка. Реестра НЕТ и `[slug]` НЕТ.** Список постов вкладки авто-генерится
  `lib/parser-fs.mjs` → `_list.generated.ts` (gitignored) на сборке (`npm run gen:lists`): добавил папку —
  список пересобрался сам, ноль правок существующих файлов; удалил папку — документ исчез без сирот. Новая
  вкладка = одна строка в `COLLECTIONS` (`typeModule: './_lib/post'`). Доменные хелперы вкладки живут в
  `app/[lang]/<tab>/_lib` (ФУНКЦИИ + контракты типов), а UI-строки — в `<tab>/_data` (ДАННЫЕ,
  `get<Tab>Ui(lang)`). Полная архитектура с исходниками — [`content-engine.md`](./content-engine.md).

> **🔒 Один пост — на все языки; `slug` = языко-независимый идентификатор.** Единица контента — ОДИН
> пост, охватывающий все языки, а не по посту на язык. `slug` выбирается ОДИН раз из БАЗОВОГО
> (английского) заголовка и переиспользуется: `/en/<tab>/<slug>` и `/es/<tab>/<slug>` — тот же `<slug>`,
> отличается лишь языковой префикс. Перевод — ячейка `<lang>.ts` в ТОЙ ЖЕ папке, НИКОГДА не второй пост.
> Не слугифицировать переведённый заголовок; не создавать пост по разу на язык. Создание всегда по
> английскому идентификатору; перевод — отдельный путь (`owner_content_translate_pending`), он пишет в
> ячейку, а не создаёт пост.

## §2. Перевод «по ключам», а не «всё или ничего» (deep-merge)

Резолвер документа и оболочка интерфейса отдают язык как **рекурсивное слияние базы с оверрайдом**, а не
«взять язык целиком ИЛИ откатиться на базу целиком». Недостающий ключ берётся из `en` **поштучно**.

- **Железное правило:** **массивы замещаются целиком, никогда не сливаются поэлементно.** Частичный перевод
  списка почти всегда значит «перевести весь список или унаследовать базовый», а не смешать языки в одном.
- **Ценность для масштаба:** 81-й язык можно завести **с одним переведённым полем** — остальное корректно
  отрендерится из базы, без блокировки «переведи всё или не публикуй». Приоритетные языки получают полный
  перевод, длинный хвост живёт в режиме base-fallback по умолчанию.

## §3. Никакого выбора по языку в коде страниц

**Запрещено** в страницах/компонентах: `язык === 'xx' ? X : Y`, `const isXx = lang === 'xx'`, ручной
маппинг локали. Вместо этого — три приёма:

1. **Надпись интерфейса (хром: крошки/«назад»/заголовки)** → ключ в `_data`-хроме вкладки
   (`get<Tab>Ui(lang)`, файлы `_data/{en,ru}.ts`), не строка в `_components`. Строку со ссылкой в середине —
   бить на `…Pre` / `…Link` / `…Post`, чтобы порядок слов оставался переводимым.
2. **«По языку нужен другой блок/компонент»** → **дискриминатор-поле в данных** + реестр компонентов.
   Страница читает значение из контента и берёт компонент из реестра. Новый непереведённый язык наследует
   значение базы через deep-merge (§2) автоматически — без ветки на каждый язык.
3. **Дата/число** → отдавать `lang` напрямую в `toLocaleDateString(lang, …)` (голый ISO-639-1 — валидный
   BCP-47 tag), не маппить вручную (это ещё и убирает класс багов «дата RU-страницы в английском формате»).

**Защита от регрессии.** Документация одной фиксации не удержит линию при росте числа страниц и
контрибьюторов (включая ИИ-агентов). Поставьте lint-правило, ловящее сравнение с локаль-литералом
(`=== 'ru'` / `=== 'en'`) в слое страниц; легитимные серверные нормализации — в сокращающемся allowlist.

## §3.5. Безопасность языков — приложение НЕ падает из-за языка (прививка, шаг 149)

Два железных инварианта на уровне функций (модель может ошибиться — приложение не должно падать):

1. **Один авторитет языков, одна переменная.** Набор языков задаётся ЕДИНСТВЕННЫМ источником
   `config/translations/translations.config.ts` через ЕДИНСТВЕННУЮ env `NEXT_PUBLIC_SUPPORTED_LANGUAGES`.
   Движок (`lib/languages.ts`) ре-экспортит его — роутинг, switcher, SEO/`hreflang` и контент резолвят
   один и тот же список. Второго авторитета (второй переменной, второго `SUPPORTED_LANGUAGES`) быть НЕ
   может: расхождение = язык «поддержан здесь, неизвестен там» → 500.
2. **Лишний язык игнорируется, не роняет.** Контент на языке вне набора — это мёртвый оверрайд
   (`resolveEntry` читает его только при совпадении языка; маршрут `[lang]` 404-ит несписочный сегмент).
   Дата форматируется через `formatLocalizedDate` (`lib/i18n/format-date`) — невалидный тег даёт fallback,
   не `RangeError`. Любой оставшийся throw в `[lang]`-дереве ловит **error-граница** (`app/[lang]/error.tsx`
   + `app/global-error.tsx`) — страница деградирует, а не отдаёт голый Internal Server Error.

**Следствие для генерации (агенту):** генерируй контент СТРОГО в пределах набора языков (читается из
`NEXT_PUBLIC_SUPPORTED_LANGUAGES` в `app/.env.local` — без API). Композер (`compose-frozen-template`) по
умолчанию берёт реальный набор слота, а не хардкод — он НЕ печёт язык, которого нет в приложении.

## §4. Статическая генерация и SEO/GEO

> **Канон (обязательно): «лучше ничего, чем динамика».** Динамические страницы запрещены, кроме
> architect-only служебных (исключение архитектора). Публичные `[lang]`-страницы — только SSG/ISR.
> **Капкан:** корневой `export const dynamic = "force-dynamic"` на `app/layout.tsx` делает динамическим
> ВЕСЬ субдерево (включая контент) — никогда так не делать; Site Settings отражать через ISR
> (`revalidate`), не через корневую динамику. Полный канон и how-to —
> [`STATIC-FIRST.md`](../../STATIC-FIRST.md) + [`static-first.md`](./static-first.md). Эталон —
> `Documents/fractera/22slots-main/app`.

- На малом объёме — полная статическая генерация по всем существующим языкам (быстро, корректно). Когда
  (число языков × число документов) раздувает сборку — переход на ISR **без потери статики**:
  `dynamicParams = true` (дорендер на лету для языков вне списка — резолвер уже даёт base-fallback) +
  `revalidate = N` (кэш после первого запроса). Список параметров — только по **реально существующим**
  языковым файлам, не по всем поддерживаемым языкам.
- **Каждый язык — тот же `slug` под своим языковым префиксом** (`/en/<slug>`, `/es/<slug>`) с
  корректными `hreflang`/`alternates` и своей SEO-поверхностью (title / description / keywords /
  подзаголовок). Перевод подаётся под своим углом, а не дословной калькой — чтобы поисковик видел
  разные языковые версии, а не дубль. Это НЕ отдельный пост и НЕ переведённый slug.
- **Машиночитаемые индексы для ИИ** (`llms.txt` / `llms-full.txt`, sitemap) обновляются вместе с
  публикацией — иначе ИИ-краулеры и поисковые агенты документ не обнаружат. На больших объёмах эти индексы
  генерируются из авто-сгенерированного списка (`_list.generated.ts` / `parser-fs`), а не пополняются вручную.
- **🔒 Изображение страницы → сниппет ВСЕГДА.** Если у страницы/документа (новость / блог / контент-страница)
  есть собственное главное изображение, оно ОБЯЗАНО уходить в сниппет (`og:image` + Twitter
  `summary_large_image`): передавать его в `constructMetadata({ image })`, а НЕ полагаться только на глобальный
  `cfg.images.ogImage`. Нельзя оставлять сниппет без картинки страницы, когда картинка есть. Абсолютный URL
  обеспечивает `metadataBase`; соц-краулеры (Telegram / Facebook / LinkedIn) относительные пути игнорируют.
  Обновить превью в Telegram — отправить URL боту `@WebpageBot`. (Введено 2026-06-21.)

## §5. Навык `create-multilingual-content-entry`

Чтобы подход был частью архитектуры, а не «знанием в голове», его несёт навык агента. Он самодостаточен
(работает у любого агента, без Hermes и памяти) и делает ровно одно: **заводит мультиязычный документ
правильно** — co-located папка маршрута `app/[lang]/<tab>/<slug>/`, полная база `en`, частичные оверрайды,
авто-обнаружение (без реестра и `[slug]`), проверка на отсутствие языковых костылей. Канон —
`.agents/skills/create-multilingual-content-entry/SKILL.md` + копии в каждую агент-папку (самодостаточность
§0 методологии). Имя по стандарту именования (4–6 слов, человекочитаемо — см. `development-methodology.md`).
